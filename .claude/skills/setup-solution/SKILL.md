description: Set up the solution structure with a solution.yaml and assets folder, each containing an asset with an asset.yaml file. Used to create a solution, to create an asset and to set up a multi-asset solution.

Sets up `solution.yaml` and `assets/` folder inside the current working directory. Each asset has its own subfolder with an `asset.yaml`.

**All files created by this skill MUST be inside the current directory.**

## Instructions

### Schema Reference

**solution.yaml Structure:**
```yaml
apiVersion: solution.sap/v1
kind: Solution

metadata:
  name: <solution-name>                 # Unique name for your solution. When generating a name, append a 5-char UUID fragment (e.g. my-solution-3f8a1): node -e "process.stdout.write(require('crypto').randomBytes(3).toString('hex').slice(0,5))"
  version: 1.0.0                        # Solution version
  description: <solution-description>   # Brief description of the solution

assets:                                 # Assets to include
  - ref: ./assets/<asset-name>/asset.yaml
```

**asset.yaml Structure:**
```yaml
apiVersion: asset.sap/v1
kind: Asset

metadata:                 # Asset identity - modifying the name after deployment will cause breaking changes (such as URLs of the deployed asset)
  name: <asset-name>      # Identifying name using kebab-case: lowercase letters (a-z), numbers (0-9), and hyphens (-) only. NO spaces, NO uppercase, NO underscores. Max 63 chars. Examples: "my-agent", "greeting-bot-v2"
  version: 1.0.0          # Semantic version of this asset

type: <asset-type>        # Asset type
 
sourceRoot: "workflows"   # Root directory where workflow files are located
 
# Artifacts provided by this asset
provides:
  <artifact type specific content>
```

### Asset Types

The last segment of the asset name  determines the asset type. For example:
- `purchase-order-approval-agent` → type `agent`
- `invoice-processing-n8n-workflow` → type `n8n-workflow`
- `s4-dispute-mcp-server` → type `mcp-server`

Supported types:
- `base-ui`: Frontend web applications (Next.js, React)
- `cap-app`: SAP CAP Application
- `agent`: AI agent services (LangChain, custom)
- `agent-extension`: Agent extension packages that extend a base agent with additional capabilities
- `domain-model-extension`: Domain model extensions that add custom fields to existing CAP entities. **This asset type contains `asset.yaml` and `extension.yaml` files.** The `asset.yaml` references the `extension.yaml` descriptor file via the `descriptorFile` property.
- `n8n-workflow`: N8N workflow automation
- `vercel-app`: Vercel-specific deployments
- `mcp-server`: MCP Server metadata files (i.e. translation files for API-to-MCP tool mapping). **This asset type contains only the following files: `asset.yaml`, `translation.json`, `.tool-list.json`, and `api-spec.<extension>` — NO other files: NO server implementation, NO Dockerfile, NO source code.** The translation file is generated by the `mcp-translation-file` skill and copied into the asset folder. The original API spec file used to generate the translation should also be copied for reference. The asset.yaml is generated by this skill and must reference the translation file in its `metadata.translation` field.
- `data-product`: Data product definition *(Only if `skills/data-product-generation/SKILL.md` exists; otherwise inform: "Data product asset unavailable - skill not found")*

**Data Product Availability Check:**
- If user query includes Data Product and `skills/data-product-generation/SKILL.md` exists, proceed with data product asset creation
- If skill file missing and user requested data products: inform "Data product generation unavailable."

**Note**: Each asset type has a detailed JSON Schema in this skill's folder (`skills/setup-solution/assets/{type}.json`) with:
- Complete property definitions (100+ properties for agents)
- Strict validation rules (patterns, enums, min/max values)
- Detailed descriptions for every field
- Default values and constraints
- Examples: agent.json includes agentCard, models, secrets, HPA, probes, resources, virtualService, etc.
- domain-model-extension.json includes entities, fields, types, translations, and appNamespace

### Multi-Component Pattern (AppRouter + Next.js)

**IMPORTANT**: For AppRouter + Next.js setups:

1. **Order matters**: AppRouter MUST be first component
2. **Destinations**: Use environment variable, NOT xs-app.json
3. **Ports**: AppRouter = 5000, Next.js = 3000

```yaml
components:
  - name: appRouter
    buildPath: router
    port: 5000
    env:
      destinations: '[{"name":"backend","url":"http://127.0.0.1:3000","forwardAuthToken":false}]'

  - name: server
    buildPath: src
    port: 3000
    env:
      PORT: "3000"
```

**xs-app.json** (minimal, NO destinations block):
```json
{
  "welcomeFile": "/",
  "authenticationMethod": "none",
  "sessionTimeout": 60,
  "routes": [
    {
      "source": "^/(.*)$",
      "target": "$1",
      "destination": "backend",
      "authenticationType": "none"
    }
  ],
  "compression": {"enabled": true},
  "websockets": {"enabled": true}
}
```

#### Example 3: New MCP Server Asset
```yaml
apiVersion: asset.sap/v1
kind: Asset
type: mcp-translation
metadata:
  name: s4-dispute-mcp-server
  ordId: customer.mcpbuilder.s4:apiResource:full-stack-app-7c2e4_s4-dispute-mcp-server:v1
  version: "1.0.0"
  translation: mcp-translation/translation.json
```

The MCP server asset folder must **only** contain the following files:
```
assets/s4-dispute-mcp-server/
├── asset.yaml                # as above
└── mcp-translation/
    ├── api-spec.<extension>  # original API spec
    ├── translation.json      # Translation file (generated by mcp-translation-file skill)
    └── .tool-list.json       # Tool list file (generated by mcp-translation-file skill)
```
Do NOT create a Dockerfile, source code, or any other files for new MCP server assets.

**File sources:**
- Translation files are generated by the `mcp-translation-file` skill (via the `generate_mcp_translation` webagent tool).
- API Spec files are the original API specs (downloaded during API discovery phase) that were used as input to the `mcp-translation-file` skill. They are still needed to ensure we can (re)generate the MCP Server Card during deployment.
- The `asset.yaml` file is generated by this skill and must reference the translation file in its `metadata.translation` field.

**MCP Server ORD ID Convention:**

Use the MCP server asset ORD ID format defined in the shared guide:
```
customer.mcpbuilder.<system>:apiResource:<solution-name>_<asset-name>:<version>
```

Example:

```yaml
metadata:
  name: s4-dispute-mcp-server
  ordId: customer.mcpbuilder.s4:apiResource:full-stack-app-7c2e4_s4-dispute-mcp-server:v1
```

Where the parts are derived as follows:
- `<system>` -> source system from `translation.json` `target.ordId` (for example, `sap.s4:...` -> `s4`)
- `<solution-name>` -> `solution.yaml` `metadata.name`
- `<asset-name>` -> MCP server `asset.yaml` `metadata.name`
- `<version>` -> semantic asset revision with the required `v` prefix (`v1`, `v2`, ...)

> **_Important note:_** For existing MCP Servers (registered in UMS), this entire section doesn't apply. They are only referenced in the agent's `asset.yaml` via their known ORD IDs, but no asset creation or file generation is needed in this case.

#### Example 4: Domain Model Extension Asset
```yaml
apiVersion: asset.sap/v1
kind: Asset

metadata:
  name: bookshop-addresses-extension
  version: 1.0.0
  description: "Extends my.bookshop.Addresses entity with testField1"

type: domain-model-extension

sourceRoot: "."

provides:
  extensions:
    - descriptorFile: extension.yaml
      name: bookshop-addresses-extension
```

The domain model extension asset folder contains `asset.yaml` and `extension.yaml`:
```
assets/bookshop-addresses-extension/
├── asset.yaml          # as above
└── extension.yaml      # extension descriptor with entities, fields, translations
```

**extension.yaml** structure:
```yaml
# Opaque content for domain model extension asset
appNameSpace: sample-app
extensions:
  - extensionProject:
      name: BOOKSHOP-ADDRESSES-EXTENSION
      description: Extends my.bookshop.Addresses entity with testField1
      customAttributes: []
    types: []
    entities:
      - entityName: my.bookshop.Addresses
        displayName: Addresses
        isParametric: false
        fields:
          - fieldName: testField1
            isKey: false
            type:
              name: String
              isExtended: false
              isBase: true
            fieldLabel: Test Field1
    translations:
      - entityName: my.bookshop.Addresses
        locale: de
        translationKey: '{i18n>testField1}'
        translationValue: Testfeld 1
```

After the files are generated, the user can optionally deploy the extension via the `deploy_domain_model_extension` tool (deployment is on-demand only, requires explicit user approval).

#### Example 5: Agent Extension Asset
```yaml
apiVersion: asset.sap/v1
kind: Asset

metadata:
  name: employee-onboarding-tools
  version: 1.0.0

type: agent-extension

sourceRoot: "."

provides:
  extensions:
    - descriptorFile: extension.yaml
      name: employee-onboarding-tools

requires:
  - name: employee-onboarding-agent
    type: agent
    version: "1.0.0"
    ordId: "sap.ai:agent:employee-onboarding:v1"
```

The agent extension asset folder contains `asset.yaml` and `extension.yaml`:
```
assets/employee-onboarding-tools/
├── asset.yaml          # as above — MUST include requires with agent reference
└── extension.yaml      # extension descriptor (tools, hooks, instructions)
```

**CRITICAL**: The `requires` array MUST include at least one entry with `type: agent` referencing the base agent's `ordId` and `version`. Without this, the platform cannot resolve the agent dependency.

#### Example 6: Multi-Asset Solution (Agent + MCP Servers)
```yaml
apiVersion: solution.sap/v1
kind: Solution
metadata:
  name: full-stack-app-7c2e4
  version: "1.0.0"
  description: "Agent + MCP Server example"
assets:
  - ref: ./assets/agent/asset.yaml
  - ref: ./assets/s4-dispute-mcp-server/asset.yaml
```

When an agent needs access to tools defined in MCP server translation files, add the MCP server ORD IDs to the agent's `asset.yaml` under `requires`:
```yaml
# In assets/agent/asset.yaml:
apiVersion: asset.sap/v1
kind: Asset
type: agent
metadata:
  name: my-agent
# ... other agent config ...
requires:
  - name: S4_DISPUTE_MCP
    kind: mcp-server
    ordId: customer.mcpbuilder.s4:apiResource:full-stack-app-7c2e4_s4-dispute-mcp-server:v1
```
For newly created MCP Server assets, the ORD IDs in the `requires` field of the agent's `asset.yaml` must match the ORD IDs of the MCP server assets defined in their respective `asset.yaml` files.
For existing MCP Servers with known ORD IDs, the ORD IDs in the `requires` field of the agent's `asset.yaml` must match the known ORD IDs of the existing MCP servers.

#### Example 7: Multi-Asset Solution (Agent + n8n Workflow)

When a solution includes both an AI agent and an n8n workflow, the n8n workflow asset **MUST** also be referenced in `solution.yaml`. The n8n asset is **always named `n8n`** and **always has type `n8n-workflow`**.

```yaml
# solution.yaml
apiVersion: solution.sap/v1
kind: Solution
metadata:
  name: invoice-approval-3a9f2
  version: "1.0.0"
  description: "Invoice approval with AI agent and n8n workflow"
assets:
  - ref: ./assets/invoice-approval-agent/asset.yaml
  - ref: ./assets/n8n/asset.yaml
```

```yaml
# assets/n8n/asset.yaml
apiVersion: asset.sap/v1
kind: Asset
metadata:
  name: n8n
  version: "1.0.0"
type: n8n-workflow
sourceRoot: "workflows"
provides:
  workflows:
    - definitionFile: invoice-approval.n8n.json
      name: invoice-approval
```

> `sourceRoot: "workflows"` means each `definitionFile` is resolved relative to `assets/n8n/workflows/`. The workflow file above is therefore stored at `assets/n8n/workflows/invoice-approval.n8n.json`.

The folder structure for this solution:
```
./
├── solution.yaml
└── assets/
    ├── invoice-approval-agent/
    │   ├── asset.yaml
    │   └── ...
    └── n8n/
        ├── asset.yaml
        └── workflows/
            └── invoice-approval.n8n.json
```

### Best Practices

1. **Do NOT add `container.env` to agent assets** — it causes schema validation errors on the platform. Set environment variables via `ENV` instructions in the Dockerfile instead. The `env` block is only valid for `base-ui` / `service` component definitions.

2. **Keep agent `asset.yaml` minimal** — omit `resources` and `hpa` blocks unless you have a specific reason to override platform defaults. Extra fields have caused schema validation failures on the platform.

3. **Include health probes** (use `/.well-known/agent.json` for A2A agents):
   ```yaml
   probes:
     startup:
       path: /.well-known/agent.json
       periodSeconds: 5
       timeoutSeconds: 3
       failureThreshold: 18
     liveness:
       path: /.well-known/agent.json
       initialDelaySeconds: 15
       periodSeconds: 10
       timeoutSeconds: 5
       failureThreshold: 3
     readiness:
       path: /.well-known/agent.json
       initialDelaySeconds: 5
       periodSeconds: 5
       timeoutSeconds: 3
       failureThreshold: 3
   ```

4. **Use environment variables** for configuration (UI/service assets only — see note 1 for agents):
   ```yaml
   env:
     PORT: "3000"
     LOG_LEVEL: info
   ```

5. **For AppRouter assets**, NEVER include destinations in xs-app.json - use env var instead

6. **Standard ports**:
  - Python agents: 5000
  - Node.js: 3000
  - Java: 8080
  - AppRouter: 5000

7. **Required folder structure** — always use this layout:
  ```
  ./
  ├── solution.yaml             # refs ./assets/<asset-name>/asset.yaml
  └── assets/
      └── <asset-name>/
          ├── asset.yaml
          └── ...                 # any other asset files that are needed (e.g. Dockerfile, source code, xs-app.json, translation file, etc.).
  ```
  When applicable:
    - The `Dockerfile` MUST be co-located with `asset.yaml` (i.e. inside `assets/<asset-name>/`).
    - The `container.buildPath` in `asset.yaml` must be `.` (relative to asset.yaml location).

### Common Issues

**AppRouter Fails with "unknown destination backend"**
- ✅ Remove `destinations` block from xs-app.json
- ✅ Add `destinations` environment variable in asset.yaml

**Service exposes wrong port**
- ✅ Ensure AppRouter is first component in list

**Build fails**
- ✅ Check buildPath points to correct directory
- ✅ Verify package.json exists

### Workflow

When user requests YAML creation:

1. All solution files (i.e. `solution.yaml` and `assets/` folder) **MUST** live inside the current working directory. Create the assets folder if it does not exist (`mkdir -p assets`).

   **Generate a UUID fragment** for the solution name before creating `solution.yaml`:
   ```
   node -e "process.stdout.write(require('crypto').randomBytes(3).toString('hex').slice(0,5))"
   ```
   Append the output to the solution name (e.g. `my-solution-3f8a1`). This ensures uniqueness across deployments. Users may rename it afterwards.

2. **Ask clarifying questions if unclear**:
   - What type of asset? (UI, agent, service, mcp-server)
   - Single or multi-container? (not applicable for mcp-server)
   - What framework? (Next.js, Python, etc.) (not applicable for mcp-server)

3. **Consult JSON schemas for validation rules**:
   - Check `assets/solution-schema.json` for solution structure
   - Check `assets/{type}.json` for asset-specific fields
   - Note required fields, patterns (e.g., DNS-compatible names), and enums

4. **Create appropriate structure inside `assets/`**:
   - Use single-container for simple apps
   - Use multi-component for AppRouter patterns
   - For `mcp-server` assets: create only `asset.yaml` + copy files generated by the `mcp-translation-file` skill — nothing else
   - For `domain-model-extension` assets: create only `asset.yaml` — nothing else (deployment via `deploy_domain_model_extension` tool is optional and on-demand, after asset.yaml is generated)
   - For `agent-extension` assets: create `asset.yaml` and `extension.yaml`. The `asset.yaml` MUST include a `requires` entry with `type: agent` referencing the base agent's `ordId` and `version` (see Example 5)

5. **Handle MCP server assets** (only do this if translation files exist in `specification/<asset-name>/mcps/`, otherwise ignore this step):
   - For each subfolder inside `specification/<asset-name>/mcps/`:
     a. Create the asset folder: `assets/{mcp-server-name}/` and the mcp-translation subfolder: `assets/{mcp-server-name}/mcp-translation/`
     b. Copy the files to the asset folder: `cp specification/<asset-name>/mcps/{name}/* assets/{mcp-server-name}/mcp-translation/`
     c. Create `assets/{mcp-server-name}/asset.yaml` following the `mcp-server` schema (see Example 3)
     d. Register the asset in `solution.yaml` under `assets`, i.e.: `- ref: ./assets/{mcp-server-name}/asset.yaml`
   - If the solution includes an agent asset that uses these APIs, add the MCP server ORD IDs to the agent's `asset.yaml` under `requires` (see Examples 2 and 6)
   - Do NOT create a Dockerfile, source code, or server implementation for MCP server assets

6. **Include all required fields**:
   - metadata.name (solution) - must match pattern `^[a-z0-9]([-a-z0-9]*[a-z0-9])?$`
   - metadata.id (asset) - DNS-compatible
   - type - must match available schemas
   - container OR components (not applicable for mcp-server assets)

7. **Add best practices**:
   - Resource limits and environment variables (UI/service assets only — omit for agent assets, see Best Practices)
   - Health probes

8. **Validate**:
   - Check YAML syntax
   - Verify all required fields present (per JSON schema)
   - Ensure paths are correct and all paths are relative to `solution.yaml`
   - Validate against patterns and constraints

## Important Notes

- Schema version: v3.0.0
- All paths in `solution.yaml` are relative to the `solution.yaml` location
- Field names are case-sensitive
- Use lowercase-with-hyphens for IDs
- The `Dockerfile` must live inside the asset folder (co-located with `asset.yaml`), not at the solution root
- For A2A agents, always use `/.well-known/agent.json` as the probe path (startup + liveness + readiness)
- MCP server asset folders must contain ONLY `asset.yaml` + files generated by the `mcp-translation-file` skill — no Dockerfile, no source code, no server implementation
- Domain model extension assets contain `asset.yaml` and `extension.yaml` files; deployment via `deploy_domain_model_extension` tool is optional and on-demand (requires user approval after files are generated)
- The n8n workflow asset is **always named `n8n`** with `type: n8n-workflow`
