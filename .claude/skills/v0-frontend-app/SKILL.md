# v0 Frontend App

Create production-quality frontend apps with Vercel v0 by composing a strong, structured prompt and then driving the gateway tools (`create_app`, `send_app_message`, `get_app_preview`) — all pinned to `provider="v0"`.

This skill is the right tool when the user wants a *UI artifact*: a web app, dashboard, portal, landing page, internal tool, marketing page, or React/Next.js component. It is **not** for backend services, AI agents, n8n workflows, or BTP/CAP extensions.

## When to Use This Skill

**Both** conditions must be met:

1. The Vercel gateway tools (`create_app`, `send_app_message`, `get_app_preview`) are available in the current environment.
2. The user **explicitly** asks to create an app with **Vercel** or **v0**.

Do **not** use this skill when:

- The user asks for a generic \"frontend app\", \"web app\", \"dashboard\", etc. without mentioning Vercel or v0 → use the `intent-analysis` → `product-requirements-document` → `specification` chain instead. (When the user *does* mention v0 or Vercel, this skill still runs `intent-analysis` and `product-requirements-document` first — see Step 1.)
- The user wants an **AI agent** → use `sap-agent-bootstrap` / `create-agent-extension`.
- The user wants a **workflow / automation** → use `n8n-workflow`.
- The user wants a **BTP extension / CAP service / Fiori app** → use the `intent-analysis` → `product-requirements-document` → `specification` chain.
- The user wants **simple static files or scripts** → use the file tools directly.

## Prerequisites

- The host agent (e.g. webagent) must expose the **gateway tools**: `create_app`, `send_app_message`, `get_app_preview`. These ship with `@sap/webagent`'s `gatewayToolModule`. If they are not registered, stop and tell the user the host environment does not support v0 generation.
- A backend reachable at the configured `apiBaseUrl` that proxies to the v0 gateway (`POST /gateway/v0/create-app`, `POST /gateway/v0/send-message`, `GET /gateway/v0/preview/:projectId`, `GET /gateway/v0/files/:projectId`).
- The user has accepted that the app will be generated by an external provider (Vercel v0) and that files will be synced back into the current webagent project.

## Tools

Always pass `provider="v0"`. Do not ask the user which provider to use — this skill is v0-only.

| Tool | When to call | Notes |
|------|--------------|-------|
| `create_app` | Once per asset, **after** `intent-analysis` and `product-requirements-document` have run, to scaffold the initial app. | Required: `provider="v0"`, `prompt`, **`asset_path`** (the `assets/<asset-name>` directory `setup-solution` scaffolded), **`prd_files`** (must include `product-requirements-document.md`). Optional: `project_name`, `solution_id`. On success, the webagent project context is switched to the new v0 project. |
| `send_app_message` | For every follow-up change the user requests. | Required: `provider="v0"`, `solution_id`, **`asset_path`**, `message`. After each call, files are synced back automatically (best-effort). |
| `get_app_preview` | When the user asks "where can I see it?", or after a meaningful change, or to confirm the preview URL. | Required: `provider="v0"`, `solution_id`, **`asset_path`**. Returns `null` if no preview is available yet — surface that clearly. |

Do **not** use `write_file` / `update_file` / `execute_command` to hand-edit a v0-generated app. v0 owns the source of truth — round-trip changes through `send_app_message` so the provider stays in sync. The only exception is small post-processing (e.g. patching a config the user explicitly asked for that v0 cannot do).

### Choosing the right tool: `create_app` vs `send_app_message`

Always check first whether the current solution already has a v0 app:

- **Asset already has a v0 app** (the asset's `assets/<asset-name>/` directory has v0-synced files, or `get_app_preview` for that asset returns a non-null URL) → use `send_app_message` with the same `asset_path`. Do **not** call `create_app` again for that asset — it would attach a *second* v0 project to the same asset.
- **Asset has no v0 app yet** → use `create_app` with that asset's `asset_path`.

`send_app_message` is **only** for changes the user explicitly asks for *after* the initial scaffold. Specifically:

- **Do not** call `send_app_message` immediately after `create_app` with the same instructions, the PRD content, or a paraphrase of either. `create_app` already passed the PRD to v0 via `prd_files`; re-sending it confuses v0 and burns through the 1000-character per-message budget.
- **Do not** "kick off" v0 with `send_app_message` after `create_app` — the scaffolding from `create_app` is already in flight. Wait for the user's first follow-up request and send only that.
- **Do** call `send_app_message` when the user asks for a specific change (add a component, change a color, rewrite a piece of copy, etc.).

A single solution can host **multiple** v0 apps as long as each lives under a distinct `assets/<asset-name>/` directory. The natural key for every gateway call is `(solution_id, asset_path)` — the same shape every other artifact uses (cap-app, n8n-workflow, mcp-server, …).

### `solution_id` parameter — what to pass

For `send_app_message`, `get_app_preview`, and `get_app_files`: **always pass the
solution id of the active project** alongside `asset_path`. The pair forms the
natural key for the partner mapping; either alone is ambiguous.

For `create_app`:

Default behaviour: **omit `solution_id`**. The host (`@sap/webagent`) auto-fills it from the current project context, so the new v0 app attaches to the solution the user is already working in. This is what you want 99% of the time.

Pass `solution_id` explicitly only when:

- The user has explicitly asked you to scaffold a v0 app for a *different* solution they own (rare).
- You are running outside any project context (very rare — most agent invocations are inside a project).

Never invent or guess a UUID. If unsure, omit the parameter and let the host default kick in.

### Attaching a PRD with `prd_files` (REQUIRED)

`create_app`'s `prompt` is forwarded to v0's `description` field, which v0 caps at **1000 characters**. A real user request — once stack, screens, data, and design constraints are spelled out — will almost always overflow that limit. To keep `prompt` short while still giving v0 full context, this skill **requires** the `prd_files` parameter on every `create_app` call.

The PRD is produced by Step 1 of the workflow (`intent-analysis` → `product-requirements-document`). After Step 1 completes, `product-requirements-document.md` exists in the current working directory and is the file you attach.

Keep `prompt` short and directive — typical shape: `"Build the application described in the attached PRD."` — v0 will treat the files as the source of truth and pick the appropriate stack based on what the PRD describes. Only pin a stack in the `prompt` if the PRD does not specify one and the user has a hard requirement.

Pass `prd_files` as an array of paths relative to the solution version:

```
create_app({
  provider: "v0",
  prompt: "Build the application described in the attached PRD.",
  asset_path: "assets/<asset-name>",
  prd_files: ["product-requirements-document.md"],
})
```

You may include `intent.md` alongside the PRD if it adds context not captured in the PRD (e.g. the original business challenge wording): `prd_files: ["product-requirements-document.md", "intent.md"]`.

Do **not** call `create_app` without `prd_files`. If you find yourself wanting to, go back and run Step 1 first.

## Workflow

### **MANDATORY: Setup the Solution**

Before calling `create_app`, run the `setup-solution` skill to scaffold `solution.yaml` and `assets/<asset-name>/asset.yaml`. The v0 app must live as a `vercel-app` asset inside the solution so that Joule Studio runtime's deployer can pick it up — Joule Studio runtime's deployer supports `vercel-app` natively.

The final `assets/<asset-name>/asset.yaml` MUST follow the template provided in [./assets/asset.yaml](./assets/asset.yaml) (co-located with this SKILL.md). **Copy that template verbatim and replace only the placeholders** (`<asset-name>`) — do NOT hand-write the file from memory or derive its shape from naming conventions. The template is the canonical `vercel-app` schema (also formalized at [`../setup-solution/assets/vercel-app.json`](../setup-solution/assets/vercel-app.json)).

Hard rules — these are the exact mistakes that break artifact recognition, so do not deviate:

- **`type` MUST be exactly `vercel-app`.** It is NOT `vercel-v0`, `v0`, or `frontend`. Any other value makes the backend fail to recognize the asset (the Solution Overview shows "no artifacts").
- **Use the `apiVersion` / `kind` / `metadata` / `container` structure from the template.** Do NOT write a flat `name:`/`type:`/`description:`/`label:` file — those top-level keys are not part of the schema.
- **`container.buildPath: .`** — v0-synced files land at the root of the asset directory (`assets/<asset-name>/app/...`, `assets/<asset-name>/package.json`, …). The deployer reads from the asset directory root.
- **`container.port: 5000`** — the port the v0 Next.js app listens on. Do not change unless the user has a hard requirement.

In `solution.yaml`, reference the asset with a **`ref`** entry (NOT `path`/`name`):

```yaml
assets:
  - ref: ./assets/<asset-name>/asset.yaml
```

If `setup-solution` produces a different shape (e.g. a flat file, `type: vercel-v0`, `buildPath: src`, or a `path:` entry in `solution.yaml`), correct it to match the template and the `ref` form above before proceeding.

### Step 1: Run `intent-analysis` and `product-requirements-document` (MANDATORY before generating)

Even when the user asks for a v0 app directly, **always** run the IBD analysis chain first. The v0 app must be built from a PRD — not from the raw user prompt — to keep the generated app grounded in a real spec and to keep the prompt below v0's 1000-character limit.

This step has two sub-steps. **Both are mandatory and must not be skipped**, regardless of how detailed the user's initial request was. Do not substitute your own clarifying questions for these skills.

1. **Run the `intent-analysis` skill** — produces `intent.md` capturing the business challenge, target users, and key milestones. `intent-analysis` handles all clarifying questions itself.
2. **Run the `product-requirements-document` skill** — consumes `intent.md` and produces `product-requirements-document.md` with the full feature spec, screens, data shapes, and design constraints. The PRD becomes the source of truth that v0 reads from.

Why this is mandatory:

- v0's per-prompt limit is **1000 characters**. Real user requests almost always overflow once stack, screens, and data are spelled out. Attaching the PRD via `prd_files` keeps the prompt itself small while giving v0 full context.
- A user message like "build me a reading-list tracker" is too sparse for v0 to produce a useful first scaffold. The intent → PRD chain extracts the structure v0 needs.
- Without a PRD, `create_app` will fail with `PROMPT_TOO_LONG` for any non-trivial request.

Do **not**:

- Skip to `create_app` even if the user says "just build it", "fast track", or provides what seems like a complete description. Fast-track mode applies to the IBD skills themselves (they will produce a leaner PRD); it does **not** bypass them.
- Call `ask_question` directly to gather product requirements — that is `intent-analysis`'s job.
- Build the v0 prompt from the raw user message without going through the PRD.

After both sub-steps complete, `intent.md` and `product-requirements-document.md` will exist in the current working directory and Step 3 will attach the PRD via `prd_files`.

### Step 2: Compose the v0 directive prompt

The PRD from Step 1 carries the spec (purpose, users, screens, data, design). The `prompt` field on `create_app` is now a short **directive** that tells v0 to read the PRD — not a duplicate of the PRD content. Let v0 choose the stack based on what the PRD describes; only pin a stack in the prompt when the PRD does not specify one and the user has a hard requirement.

Default directive (suitable for most v0 apps):

```
Build the application described in the attached PRD. Pick a clean, minimal
design tone unless the PRD overrides it.
```

Use only when the user or PRD has a hard stack requirement that the PRD itself does not already convey:

```
Build the application described in the attached PRD using <stack>.
<one extra sentence with non-default libraries or design overrides if needed>.
```

Keep the directive **well under 1000 characters** — typical length is 1–3 sentences. Do **not** restate purpose, users, screens, or data here; v0 reads those from `product-requirements-document.md`.

### Step 3: Create the app

Call `create_app` exactly once. Pass `asset_path` — the directory `setup-solution`
scaffolded for this v0 app (e.g. `assets/sales-dashboard-app`) — and attach the
PRD produced by Step 1 via `prd_files`. `prd_files` is **required**:

```
create_app(
  provider="v0",
  prompt=<the short directive from Step 2>,
  asset_path="assets/<asset-name>",
  prd_files=["product-requirements-document.md"],
  project_name=<optional, e.g. "sales-dashboard-app">
)
```

If `intent.md` adds context the PRD does not capture, include it too:
`prd_files=["product-requirements-document.md", "intent.md"]`.

After the call:

- The result includes the same `(solution_id, asset_path)` you passed plus (often) a `previewUrl`.
- File sync runs in the background. Once AGG finishes, the backend emits a `chatFilesync` event for `(solution_id, asset_path)` and the UI's file tree refreshes. v0 files land under `assets/<asset-name>/...`.
- Tell the user: project created, asset path, and the preview URL if present. If `previewUrl` is `null`, say so — a preview will be available shortly. They can ask for it again with `get_app_preview`.

### Step 4: Iterate with send_app_message

Every follow-up change the user asks for — and **only** changes the user explicitly asked for (see Step 4 do-not-fire rules in "Choosing the right tool" above) — goes through `send_app_message`. Pass `asset_path` and `solution_id` (= the same key you used for `create_app`) on every call:

```
send_app_message(
  provider="v0",
  solution_id=<the solution>,
  asset_path="assets/<asset-name>",
  message=<change>,
)
```

**`message` must be under 1000 characters.** v0 caps every chat message at this length and the gateway rejects oversized messages with `PROMPT_TOO_LONG`. Keep each message to a single, concrete change (one component, one screen, one refactor). If a user's request would exceed 1000 characters, split it before sending — see the "Per-message size limit" subsection below.

Examples of typical follow-ups:

- "Add a dark mode toggle in the top right"
- "Replace the bar chart with a line chart and add a 30-day moving average"
- "The pricing table needs a 4th tier called Enterprise with a 'Contact us' button"

Guidance for follow-ups:

- **Functional / structural changes** (new components, new data, new behavior, layout changes) → use `send_app_message`. v0 will edit files and the gateway will sync them back into the project.
- **Pure visual tweaks** (colors, spacing, font size) → still go through `send_app_message`, but tell the user that for fine-grained design work the v0 web UI's Design Mode is faster than prompting.
- **One change per message** when the user gives a list — it lets v0 stay focused and produces cleaner diffs. Batch only when changes are tightly coupled.
- After each call, mention how many files were synced back (`filesSynced` in the result) and the updated preview URL if changed.

#### Per-message size limit — split large changes incrementally

v0 caps a single chat message at **~1000 characters**. The gateway enforces
this and returns a typed error code when a message is too large:

```
PROMPT_TOO_LONG: message length 1842 exceeds v0 per-message limit 1000.
Split the change into smaller incremental messages.
```

When you receive `PROMPT_TOO_LONG` (or anticipate it), do **not** retry the
same prompt. Instead:

1. **Decompose** the user's request into the smallest set of independent
   changes that, applied in order, achieve the same end state. One feature,
   widget, screen, or refactor per message.
2. **Confirm the plan** with the user in one short message before starting:
   list the steps you intend to send and ask for a yes/no go-ahead.
3. **Send one step at a time** with `send_app_message`. After each call:
   - Wait for the response (don't fire all messages back-to-back; v0
     responses can mutate the same files and concurrent edits cause merge
     loss).
   - Mention which step finished and how many files were synced.
   - If a step fails, stop and surface the error to the user — do **not**
     skip ahead.
4. **Confirm completion** at the end: re-run `get_app_preview` and tell the
   user the URL plus a brief summary of what each step changed.

Example decomposition (long request → 4 incremental sends):

> User: "Replace the dashboard with: 4 KPI cards (MRR, churn, NPS, new
> users), a 30-day revenue line chart with moving average, a recent-orders
> table with filters, dark mode toggle, and brand color emerald."

```
1. Add the 4 KPI cards (MRR, churn, NPS, new users) above the existing chart.
2. Replace the bar chart with a 30-day revenue line chart with a moving-average overlay.
3. Add a recent-orders table with date/status filters under the chart.
4. Add a dark-mode toggle in the top-right header and switch the brand accent to emerald.
```

Each step should fit comfortably under the 1000-char limit on its own.

### Step 5: Share & verify the preview

- When the user asks where to see the app, or after a meaningful iteration, call `get_app_preview(provider="v0", asset_path="assets/<asset-name>")` and surface the URL.
- If `url` is `null`, tell the user the preview is not ready yet rather than making one up.

## PRD Quality Checklist

The PRD attached via `prd_files` is what drives v0. Before calling `create_app`, the PRD produced by Step 1 should answer:

- [ ] **What** is being built (app type + name)?
- [ ] **Why** (one-sentence purpose / business challenge)?
- [ ] **Who** uses it and in what context?
- [ ] **Which screens/components** with real names, not "etc."?
- [ ] **What data** is shown (real shape, mock, or placeholder)?
- [ ] **Which stack & libraries** (only if the user has a hard requirement; otherwise leave it to v0 to choose based on the PRD)?
- [ ] **Design tone & layout** (concrete, not "modern")?
- [ ] **Constraints** ("do not invent auth", "no placeholder text", etc.)?

If the PRD does not cover one of these, go back to the `product-requirements-document` skill to refine it — do not patch the gap by stuffing detail into the `create_app` prompt (it would overflow the 1000-character limit).

## Guardrails

### Communication
- Be direct. Run `intent-analysis` → `product-requirements-document` first, then build — no extra clarification round on top of what those skills already ask.
- Always echo back your understanding (1–3 lines) before calling `create_app`.
- Report `projectId` and preview URL after every state change.
- Never claim a preview URL exists if `get_app_preview` returned `null`.

### Scope
- Pin `provider="v0"` everywhere.

- Do not use this skill for backend services, agents, n8n workflows, or BTP extensions — route to the appropriate skill.

- If the user asks for something that requires real backend wiring (auth, DB, payments), call it out before generating and offer a placeholder/mock instead.

## Examples

### Example 1: Vague request → intent → PRD → build

**User:** "Build me a frontend app with v0 for tracking my reading list."

1. Run the `setup-solution` skill (if not already done) to scaffold `assets/reading-list-app/asset.yaml` with `type: vercel-app`.
2. Run the `intent-analysis` skill — it asks clarifying questions and writes `intent.md`.
3. Run the `product-requirements-document` skill — it consumes `intent.md` and writes `product-requirements-document.md`.
4. Call `create_app` with a short directive and the PRD attached:
   ```
   create_app(
     provider="v0",
     prompt="Build the application described in the attached PRD.",
     asset_path="assets/reading-list-app",
     prd_files=["product-requirements-document.md"],
     project_name="reading-list-app",
   )
   ```
5. Report the preview URL.

### Example 2: Detailed request — still goes through intent → PRD

**User:** "Build a Next.js + Tailwind + shadcn landing page for a SaaS called Pulse with v0: hero with email signup, 3-tier pricing table, FAQ accordion, footer. Brand color emerald. No placeholder copy — write real marketing copy aimed at engineering managers."

The user already gave a lot of detail, but the IBD chain still runs — it normalizes the request into the PRD format v0 reads from and keeps the prompt under the 1000-character limit.

1. Run `setup-solution` for `assets/pulse-landing`.
2. Run `intent-analysis` — fast-track mode is fine here because the user was specific; clarifying questions stay minimal.
3. Run `product-requirements-document` — the PRD captures the screens, sections, brand color, and copy guidance the user provided.
4. Call `create_app` with the PRD attached and a short directive:
   ```
   create_app(
     provider="v0",
     prompt="Scaffold a Next.js (App Router) + Tailwind + shadcn/ui landing page from the attached PRD. Use emerald as the brand accent.",
     asset_path="assets/pulse-landing",
     prd_files=["product-requirements-document.md"],
     project_name="pulse-landing",
   )
   ```
5. Report preview URL.

### Example 3: Follow-up iteration

**User (after the landing page is up):** "Add a customer logos strip below the hero, and change the pricing CTA on the top tier to 'Start free trial'."

1. Call `send_app_message(provider="v0", message="Add a customer logos strip below the hero (6 placeholder logos in a responsive grid). Change the pricing CTA on the top tier from its current text to 'Start free trial'.")`.
2. Report files synced and updated preview URL.
