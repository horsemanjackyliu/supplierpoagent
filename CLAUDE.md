# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Workspace Is

This is a **Joule Studio** solution workspace. It uses the Intent-Based Development (IBD) workflow to build SAP AI solutions — AI agents, n8n workflows, CAP applications, MCP servers, and frontend apps. All work runs through a structured chain of skills, not ad-hoc code writing.

## MCP Server

The `joule-studio` MCP server (`jl mcp start`) is the runtime backbone. It provides tools for API discovery, fit-gap analysis, solution deployment, n8n workflow management, and more. All MCP tools are pre-allowed in `.claude/settings.local.json`.

## Development Workflow (IBD Chain)

All new solution development follows this sequence — never skip steps:

1. **`intent-analysis`** skill → writes `intent.md` (business challenge, fit-gap, recommendations)
2. **`product-requirements-document`** skill → writes `product-requirements-document.md` from `intent.md`
3. **`specification`** skill → generates `specification/` folder with per-asset checklists from the PRD
4. **Execute** `specification/specification.md` → triggers `setup-solution`, then per-asset implementation skills
5. **`deploy-solution`** skill → deploys via `deploy_solution` tool (preferred) or `jl` CLI fallback

For fast-track mode, add "fast track" to the request. This skips interactive Q&A and proceeds autonomously through all phases.

To implement after a spec exists: say `execute specification/specification.md`.

## Solution Structure

```
./
├── solution.yaml                    # refs ./assets/<asset-name>/asset.yaml
├── intent.md                        # IBD artifact (intent-analysis output)
├── product-requirements-document.md # IBD artifact (PRD output)
├── deploy_result.json               # { "solution_id": "<uuid>" } after deploy
├── specification/                   # Generated by specification skill
│   ├── specification.md             # Master checklist
│   ├── guidelines*.md               # Execution rules (copied from skills)
│   └── <asset-name>/
│       ├── specification.md
│       ├── api-specs/               # Downloaded OData/REST specs
│       └── mcps/                    # Generated MCP translation artifacts
└── assets/
    └── <asset-name>/
        ├── asset.yaml
        └── ...                      # Dockerfile, source, workflows, etc.
```

The `solution.yaml` name must include a 5-char UUID fragment (e.g. `my-solution-3f8a1`). Generate with:
```bash
node -e "process.stdout.write(require('crypto').randomBytes(3).toString('hex').slice(0,5))"
```

## Asset Types and Their Constraints

| Type | Key files | Notes |
|---|---|---|
| `agent` | `asset.yaml`, `Dockerfile`, `app/` | Python, port 5000, A2A protocol |
| `n8n-workflow` | `asset.yaml`, `workflows/*.n8n.json` | Always named `n8n`, `sourceRoot: "workflows"` |
| `mcp-server` | `asset.yaml`, `mcp-translation/` | No Dockerfile/source code — translation files only |
| `base-ui` | `asset.yaml`, `xs-app.json` | AppRouter port 5000, Next.js port 3000 |
| `cap-app` | `asset.yaml`, CAP project | Node.js, port 8080 |
| `agent-extension` | `asset.yaml`, `extension.yaml` | Must include `requires` with base agent ORD ID |
| `domain-model-extension` | `asset.yaml`, `extension.yaml` | Deploy via `deploy_domain_model_extension` tool |

**Critical asset.yaml rules:**
- Agent names: kebab-case only, max 63 chars, no underscores
- Do NOT add `container.env` to agent assets (causes schema validation errors)
- Keep agent `asset.yaml` minimal — omit `resources` and `hpa` unless explicitly needed
- Always include health probes using `/.well-known/agent.json` as probe path for A2A agents
- AppRouter: NEVER put destinations in `xs-app.json` — use `env.destinations` in `asset.yaml` instead

## Agent Development (Python)

Agents use LangGraph + SAP AI Core integration. The `sap-agent-bootstrap` skill scaffolds the project.

**Critical startup order in `main.py`:**
```python
from sap_cloud_sdk.core.ai_core import set_aicore_config
from sap_cloud_sdk.core.telemetry import auto_instrument
set_aicore_config()   # MUST be first
auto_instrument()     # MUST be before any AI framework imports
# Only then: import langchain, litellm, etc.
```

**MCP tool loading must be async and lazy** — never call `get_mcp_tools()` from `__init__()` (the HTTP server won't start before the startup probe fires):
```python
async def _get_graph(self):
    if self._graph is None:
        tools = await get_mcp_tools()
        self._graph = create_agent(self.llm, tools=tools, ...)
    return self._graph
```

All imports inside `app/` must use peer-level style: `from agent import ...`, never `from app.agent import ...`.

**Tests:** `pytest` with `conftest.py` pre-configured. Run all tests:
```bash
cd assets/<agent-name> && pip install -r requirements-test.txt && pytest
```
Run a single test file:
```bash
pytest prebuilt_tests/test_structure.py -v
```

**APIs must never be called directly.** All SAP/external API interactions must go through MCP tools — no `requests`, `httpx`, OData clients, etc.

## n8n Workflow Development

Workflows are `.n8n.json` files in `assets/n8n/workflows/`. Always:
1. Look up node `type` and `typeVersion` from the catalog via `search-nodes-catalog` MCP tool — never hardcode
2. Resolve credentials via `list_credentials` MCP tool with `global: true` before writing any node
3. Webhook `path` must be `{workflow-slug}-{webhookId}` — never a plain human-readable string
4. After writing, validate with `validate-n8n-workflow` MCP tool
5. Never use `$env.VAR` expressions in workflow JSON

SAP-specific nodes have dedicated skills: `n8n-sap-task-center`, `n8n-sap-ai-core`, `n8n-sap-agent`, `n8n-sap-mcp-client`.

## CAP Application Development

When working on CAP (`cap-development` skill):
- Initialize new project: `sh ./scripts/project-init.sh <name>` from `assets/` (not bash)
- Never run `cds add sample`; never run `npm install` in `app/react-ui/` (it's a workspace)
- Before creating/modifying any `.cds` file, call `search_model` on the cap-mcp server
- Use `sap.common.CodeList` entities for status/type/category fields — never plain `String enum`
- Run tests: `npm test` (jest); start dev server: `cds watch`

## Deployment

```bash
jl login <api-endpoint>       # if not already logged in
jl solution validate          # must output "Validation passed."
jl solution build             # outputs "Archive created: <build-location>"
jl solution deploy <build-location>
jl solution status <solution_id>
```

Result is stored in `deploy_result.json`. After deploying an agent, test with:
```bash
jl a2a message "hi" --solution <solution_id> --asset <agent_asset_name>
```

Prefer the `deploy_solution` MCP tool over the CLI when available.

## API → MCP Server Workflow

When a new SAP API needs to be exposed as an MCP server:
1. `specification` skill downloads the API spec to `specification/<asset-name>/api-specs/`
2. `mcp-translation-file` skill calls `generate_mcp_translation` tool → produces `translation.json` + `.tool-list.json` in `specification/<asset-name>/mcps/<api-spec-stem>/`
3. `setup-solution` skill creates `assets/<mcp-server-name>/` with only `asset.yaml` + the translation files
4. Agent's `asset.yaml` references the new MCP server under `requires`

MCP server ORD ID format: `customer.mcpbuilder.<system>:apiResource:<solution-name>_<asset-name>:<version>`

For existing MCP servers with known ORD IDs, skip steps 1-3 and only reference them in the agent's `requires`.
